## Overview
인증이 없는 앱에서 IP 기반 rate limit을 적용하여, 단일 클라이언트의 과도한 요청으로부터 서버를 보호한다.

## 시나리오

### 1. 정상 범위 내 요청
[상황] 사용자가 1분 이내에 10회 미만의 추출 요청을 보내면
[동작] 모든 요청이 정상 처리된다

성공 기준:
- [ ] 1분 내 첫 번째 요청 -> 정상 응답 (200)
- [ ] 1분 내 10번째 요청 -> 정상 응답 (200)

### 2. Rate limit 초과
[상황] 사용자가 1분 이내에 11번째 추출 요청을 보내면
[동작] 입력 필드 하단에 "요청이 너무 많습니다. X초 후 다시 시도해주세요" 인라인 에러 텍스트가 빨간색으로 표시된다

성공 기준:
- [ ] 1분 내 11번째 요청 -> API 응답 429, 에러 메시지에 남은 초 포함
- [ ] 에러 메시지가 기존 에러 UI(인라인 에러 텍스트)와 동일한 방식으로 표시된다

### 3. Rate limit 윈도우 리셋
[상황] Rate limit에 도달한 후 1분이 경과하면
[동작] 요청이 다시 정상 처리된다

성공 기준:
- [ ] 10회 요청 후 60초 경과 -> 다음 요청이 정상 응답 (200)

### 4. Rate limit 초과 시 에러 메시지의 남은 시간
[상황] Rate limit 초과 후 30초가 경과한 시점에 다시 요청하면
[동작] "요청이 너무 많습니다. 30초 후 다시 시도해주세요"처럼 실제 남은 시간이 표시된다

성공 기준:
- [ ] 초과 후 30초 경과 시점에 요청 -> 에러 메시지에 "30초"가 포함된다

### 5. 서로 다른 IP는 독립적으로 제한
[상황] IP-A가 10회 요청을 모두 소진한 상태에서 IP-B가 요청하면
[동작] IP-B의 요청은 정상 처리된다

성공 기준:
- [ ] IP-A가 10회 소진 -> IP-B의 첫 요청은 정상 응답 (200)

## 범위

### 포함
- `POST /api/extract` 엔드포인트에 IP 기반 rate limit 적용
- 인메모리 저장소 (Map 기반)
- 429 응답에 남은 시간(초) 포함
- 기존 인라인 에러 UI를 통한 사용자 안내

### 제외
- Redis 등 외부 저장소 연동 — 현재 규모에서 불필요, 추후 필요 시 저장소만 교체 가능하도록 설계
- 글로벌(서버 전체) rate limit — IP 기반만 적용
- Rate limit 설정의 런타임 변경 — 코드 상수로 관리

## 전제 조건
- Next.js API route에서 클라이언트 IP를 얻을 수 있다 (x-forwarded-for 또는 request header)
